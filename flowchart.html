Oxalium

High-Performance Write-Ahead Log (WAL) System based on Linux io_uring

1. Project Structure

Workspace contains two crates. The core crate oxalium implements the WAL engine, and the infrastructure crate oxalium-infra provides common utilities.

oxalium (core)

  • lib.rs -- Crate root, MiMalloc global allocator
  • wal.rs -- WAL engine: WalSystem, SlabRing, Slab, RegisteredMemory, SQ/CQ threads
  • bin/pcr-2601.rs -- Benchmark binary: multi-threaded producer workload

oxalium-infra (infra)

  • convert.rs -- IntoLossy / FromLossy numeric cast traits
  • iter.rs -- distribute() work partitioning iterator
  • process.rs -- Peak memory usage (Linux /proc, Windows API)

2. System Bootstrap Flow

The benchmark binary pcr-2601 drives the entire system. It creates the target file, bootstraps the WAL, spawns producer threads, and collects metrics.

3. Slab State Machine

Each Slab is a fixed-size write buffer. It cycles through states managed by atomic CAS operations, coordinating producers, the SQ thread, and the CQ thread.

stateDiagram-v2 direction LR [*] --> NONE : reuse() NONE --> WRITING : try_claim()\nCAS success WRITING --> WRITING : prepare_write()\ncomplete_write()\n(partial fill) WRITING --> READY : mark_ready()\nslab full or\nidle-spin retire READY --> IN_FLIGHT : mark_inflight()\nSQ thread submits\nio_uring write IN_FLIGHT --> COMPLETED : mark_completed()\nCQ thread receives\ncompletion event COMPLETED --> NONE : advance_persisted_lsn()\nnotify_waiters()\nreuse() note right of NONE : Available for new writes\nowner = next cursor note right of WRITING : Producers writing concurrently\nassigned/written counters note right of READY : All bytes committed\nawaiting io_uring submit note right of IN_FLIGHT : io_uring DSYNC write pending note right of COMPLETED : I/O done, pending LSN advance
NONE - idle, awaiting claim
WRITING - producers filling data
READY - full, queued for I/O
IN_FLIGHT - io_uring write submitted
COMPLETED - I/O done, pending recycle

5. SlabRing::reserve() Detail

The core lock-free algorithm that allows thousands of concurrent producers to write into shared slabs without mutexes. Uses exponential back-off spin + tokio yield.

6. Thread Architecture

The system uses a combination of OS threads for I/O and per-core tokio runtimes for producers. The number of producer threads is num_cpus - coop, with tasks distributed evenly.

7. Memory Layout

RegisteredMemory buffers are 16MB page-aligned blocks registered with io_uring. Each is partitioned into equal-sized slabs. All slab metadata is 64-byte (cache-line) aligned to prevent false sharing.
